
### **Revised Task Prompt (Clarified Resolution & Solver Attribution)**

**Task**

You are given the raw HTML of a GitHub issue, discussion, or pull request page.
Your task is to determine whether the *original inquiry* was resolved and summarize the outcome.

You must consider **both discussion content (comments)** and **explicit repository events** shown in the HTML, such as commits, pull requests, merges, releases, milestones, or automated closure events tied to code.

---

## **Output Format**

Return a single JSON object with **exactly three fields**:

```json
{
  "solution_status": "",
  "solution_status_reason": "",
  "solvers": []
}
```

---

## **Field Definitions and Allowed Values**

### 1. `solution_status`

* `"yes"` → the original inquiry was addressed
* `"no"` → the original inquiry was not addressed

---

### 2. `solution_status_reason`

If `solution_status` is `"yes"`, choose **one** of:

* `"discussion"` → the inquiry was resolved through explanation, clarification, or guidance in comments, with no evidence of resolving code changes
* `"code change"` → the inquiry was resolved through an explicit repository change or linkage, such as a commit, pull request, merged PR, milestone, release, or automated closure tied to code

If `solution_status` is `"no"`, choose **one** of:

* `"no responder / responder ghosted"` → the inquiry received no replies and no resolving repository changes
* `"asker ghosted"` → respondents attempted to help or requested clarification, but the original asker stopped responding before resolution

---

### 3. `solvers`

* A list of **GitHub profile URLs** of the individual(s) responsible for resolving the original inquiry.
* Only include individuals who **meaningfully caused the resolution**.

---

## **Resolution Rules (Precedence)**

### **1. Explicit Code-Linked Resolution (Highest Priority)**

If the HTML shows the issue or discussion was **closed or completed due to a linked commit, pull request, merge, release, or milestone**, you **must** classify it as resolved with:

```json
"solution_status": "yes",
"solution_status_reason": "code change"
```

**Solver attribution rules for code-linked resolution:**

* Prefer commits or pull requests that:

  * explicitly close the issue (e.g., auto-close, “closed by commit”, merged PR closing the issue)
  * are directly linked in the closure event
* If multiple commits are mentioned:

  * **Exclude commits that merely reference or mention the issue** when there is a clearer resolving commit or merge
  * Only include mention-only commits **if no explicit resolving commit or merge exists**, and the surrounding context (comments or events) indicates that the mention corresponds to the actual fix

A commit or PR that closes or merges **does not require supporting discussion** to be treated as the solution.

---

### **2. Discussion-Based Resolution**

If no resolving repository change is present:

* Determine resolution based on discussion content alone
* The inquiry is resolved if comments clearly explain, answer, or guide the original asker to a solution
* The `solvers` are the commenters whose messages contain the resolving explanation

---

### **3. Unresolved Classification**

Only classify the inquiry as unresolved if **neither** of the following are present:

* a resolving repository-linked change
* a clear resolution through discussion

When unresolved:

* Use `"no responder / responder ghosted"` if there were no replies or activity
* Use `"asker ghosted"` if respondents attempted to help but the asker stopped responding before resolution

---

## **Additional Rules**

* Base your judgment **only** on information present in the provided HTML
* Do **not** classify an inquiry as unresolved solely due to a lack of comments if a resolving code-linked closure is present
* Do **not** attribute resolution to users whose actions merely mention or reference the issue when a stronger resolving action exists
* Combine **discussion context and repository events** to determine whether a mention-only commit legitimately represents the solution
* Do not include explanations, prose, or markdown outside the JSON
* Do not infer resolution beyond what is explicitly supported by the page content

---

## **Scope of Analysis**

The “original inquiry” refers strictly to the problem or request stated in the issue or discussion body at creation time.

Subsequent comments that introduce new, tangential, or recurring issues (e.g., “I still get this error on Windows 10”) must not override evidence that the original inquiry was resolved.
